Ausführen? (ja / nein / details / abbrechen)

**Keine Obergrenze für Batch-Größe.** Auch 500+ Elemente sind erlaubt; Single-Confirm reicht.

# Bulk-Operations — Archicad-Skill

Diese Datei dokumentiert, wie wir Massen-Updates und Klassifizierungen durchführen. SKILL.md lädt sie bei Bedarf.

## Inhaltsverzeichnis

1. [Das universelle Read → Filter → Group → Confirm → Apply-Pattern](#das-universelle-read--filter--group--confirm--apply-pattern)
2. [Klassifikations-Spezifika](#klassifikations-spezifika)

4. [Mehrdeutigkeits-Handling](#mehrdeutigkeits-handling)
5. [Mid-Batch-Fehlerverhalten](#mid-batch-fehlerverhalten)

## Das universelle Read → Filter → Group → Confirm → Apply-Pattern

Bei jeder Operation, die mehr als ein paar Elemente anfasst, folgen wir diesem Ablauf:

1. **READ** — Alle Elemente vom Zieltyp laden. Paginierung vollständig durchziehen. Niemals auf einer partiellen Seite operieren (Pitfall P11).
2. **FILTER** — Auf relevante Teilmenge eingrenzen (Story, Layer, Geometrie, Property).
3. **GROUP** — Pro Element die Ziel-Klassifikation oder Property ableiten. Ableitungsregeln pro Element-Typ siehe Sektion 3.
4. **CONFIRM** — Asymmetrische Sicherheit greift: User-Confirm mit Summary, `details`-Option für vollständige ID-Liste.
5. **APPLY** — Pro Element via MCP setzen. Bei Einzel-Fehler: weitermachen, am Ende vollständiger Report mit Erfolg/Fehler-Liste.

## Klassifikations-Spezifika

Klassifizierung ist ein häufiger Bulk-Auftrag und hat eigene Gotchas:

- **Klassifikations-System ist Projekt-spezifisch.** ARCHICAD-eigenes / Uniclass / OmniClass / Custom. Vor Bulk-Klassifizierung: aktives System via Warm-up Feld 7 ermitteln. **System-scoped GUID-Lookup** — sonst landen wir mit GUIDs aus einem inaktiven System und korrupten BIM-Daten (Pitfall P5).
- **Klassifikations-IDs sind GUIDs.** Mapping Klartext → GUID per Discovery-Query (Tool-Name TODO, Phase 5 verifizieren).
- **Sub-Klassen.** Manche Systeme haben hierarchische Klassen („Wand" > „Außenwand" > „Sichtmauerwerk"). Beim Mapping immer auf das gewünschte Level zielen, nicht auf den Root.

## Pro-Element-Typ-Ableitungslogik

Wie man pro Element entscheidet, welche Klasse er bekommt. Quellen kombinierbar:

- **Wand:** Layer-Name (z. B. „Wände-Außen" → Außenwand) ODER Geometrische Lage (Außenseite eines Raum-Polygons → Außenwand) ODER Verbundene Räume (Wand zwischen zwei Innenräumen → Innenwand).
- **Öffnung:** Subtype-Erkennung (Fenster-Subtype → Fenster-Klasse).
- **Stütze:** Property „tragend" / Composite-Material.
- **Library-Object:** Subtype-Pfad oder Library-Kategorie.

Konkrete Ableitungs-Algorithmen werden in Phase 5 live verifiziert und in den jeweiligen Recipes als Worked Examples dokumentiert. TODO.

## Mehrdeutigkeits-Handling

Manche Elemente passen in keine eindeutige Klasse — z. B. eine Wand grenzt sowohl an einen Innen- als auch an einen Außenraum.

**Regel: NICHT raten.** Solche Elemente werden im Confirm-Dialog SEPARAT aufgelistet, mit Markierung „unklar". User entscheidet pro Fall, oder schließt sie vom Bulk aus.

Beispiel-Confirm-Format:

```
Klassifizierung Wände:
- 84 Wände → Innenwand
- 43 Wände → Außenwand
- 5 Wände → UNKLAR (siehe Detail-Liste)

Ausführen für die 127 eindeutigen? Unklare separat behandeln? (ja / details / abbrechen)
```

## Mid-Batch-Fehlerverhalten

Wenn während APPLY ein Einzel-Element scheitert:

- **NICHT** auto-rollback (Archicad-Undo bleibt User-Werkzeug).
- **NICHT** weiterlaufen, wenn der Fehler systemisch ist (z. B. „invalid GUID" für ALLE Elemente).
- **Weiterlaufen**, wenn der Fehler einzeln ist (z. B. ein Element existiert nicht mehr).
- **Report** am Ende: „127 verarbeitet, 124 erfolgreich, 3 Fehler bei IDs X, Y, Z mit Grund A, B, C".

## Warum überhaupt?

Statisch geschriebene Doku veraltet. Ein Skill, der sich pro Session selbst erweitert, wird mit der Zeit präziser. Wir kapseln Lern-Gewinne sofort, bevor sie verloren gehen.

## Lern-Klassen und ihre Speicherorte

| Lerntyp | Wohin |
|---|---|
| Neuer MCP-Tool-Name (Discovery-Treffer) | `recipes/<typ>.md` → Discovery-Anker, oder `reference/workflow-context.md` → Warm-up-Feld |
| Neue Gotcha eines Elementtyps | `recipes/<typ>.md` → Gotchas |
| Allgemeines MCP-Verhalten (Paginierung, Fehler-Idiom) | `reference/mcp-conventions.md` |
| Geometrie-Ableitungs-Trick für Bulk-Klassifizierung | `reference/bulk-operations.md` |
| Neuer Worked Example | `recipes/<typ>.md` → Worked Examples |
| User-Präferenz | **Memory** (nicht Skill) |
| Projekt-spezifischer Fakt (Layer-Name, GUID-Liste in EINEM Projekt) | **Memory** (nicht Skill) |

Trennlinie: Skill = generisch, projekt-übergreifend. Memory = du, dein aktuelles Projekt.

## Der Reflection-Trigger

Am Ende eines Archicad-Auftrags fragen wir den User **eine** Frage:

> „Lern-Check: War da was Neues — ein Tool-Name, eine Geometrie-Stolperfalle, ein Klassifikations-Trick — das in den Skill sollte?"

Antwort-Varianten:
- `nein` → Auftrag beendet, kein Skill-Update.
- `ja, <X>` → Wir tragen X in die richtige Datei ein (siehe Lern-Klassen oben) mit Datums-Marker.
- `check selbst` → Wir rekapitulieren den Auftrag, schlagen 0–3 Updates vor, User bestätigt.

Wir verlangen NICHT, dass jeder Auftrag etwas Neues produziert. Häufig ist `nein` die korrekte Antwort.

## Format-Konventionen für neue Einträge

Jeder gelernte Eintrag bekommt einen Inline-Datums-Marker:

```markdown
- Geschwungene Wände erwarten `radius` + `arcAngle` statt `endPoint`. <!-- 2026-06-12 -->
```

**Status-Marker (optional):**
- `<!-- ÜBERPRÜFEN -->` wenn ein Eintrag wiederholt Probleme verursacht.
- Keine Markierung = bewährt.

So sehen wir auf einen Blick: was ist frisch und noch unter Beobachtung vs. was ist über Monate stabil.

## Verification-Loop bei Wiederholfehlern

- 2 Fehlanwendungen in Folge → Eintrag mit `<!-- ÜBERPRÜFEN -->` markieren, Korrekturvorschlag.
- 3 stabile Sessions ohne Probleme → Marker entfernen.

Diese Schleife verhindert, dass falsche Lern-Einträge sich verewigen (Pitfall P9 — Hallucination-Feedback-Loop).

## Grenzen — was NICHT in den Skill gehört

- **Halluzinationen / Vermutungen ohne Live-Beleg.** Jeder neue Tool-Name oder Parameter wird durch reale Ausführung im MCP validiert, bevor er in den Skill kommt.
- **Projektspezifische Strings** („Layer ‚Wände-OG' existiert"). Gehören in Memory, nicht in den Skill.
- **Einmal-Vorfälle ohne Wiederholungs-Erwartung.** Lieber gar nicht erfassen als Lärm produzieren.
- **User-persönliche Präferenzen** („User mag keine Caps in Confirm-Dialogen"). Memory, nicht Skill.
